# AI Intelligence Radar 评测规则 v0.1

## 产品意图

在指定时间窗口内，把少量、重要、可追溯的 AI 行业信号转化为知识卡片和趋势判断，同时避免重复、过期信息与来源等级混淆。

## 评分方法

每个维度使用三档评分：

| 分数 | 含义 | 产品判断 |
| :---: | --- | --- |
| `0` | 失败或严重缺失 | 不可交付 |
| `1` | 部分达标，需要人工修正 | 需要修复或实验 |
| `2` | 达到当前版本标准 | 可以进入下一环节 |

首批自动评测包含六个维度：

1. **相关性**：应该出现的趋势出现，不应该出现的趋势没有出现。
2. **证据完整性**：知识卡保留发布时间、证据段和原始链接。
3. **覆盖度**：最终输出数量符合任务预期。
4. **去重与时效**：重复内容被合并，时效窗口外内容被排除。
5. **判断价值**：知识卡包含“为什么重要”、影响评分、可信度和判断边界。
6. **过程可靠性**：卡片、趋势报告和运行计数保持一致。

单个案例默认平均分达到 `1.8` 且没有一票否决项，才算通过。

## 一票否决项

- T3 社区信号未明确标注社区层级，却被当成官方事实。
- 输出没有保留原始来源 URL。
- 编造事实、链接或证据。
- 泄露密钥、Token 或个人隐私。
- 未经确认执行发帖、评论、关注、删除等外部操作。

当前自动化只能机械检查前两项。其余项目必须通过人工抽检、工具日志和后续 Agent Trace 共同判断。

## 首批案例

- 正常案例：多来源趋势与重复信号。
- 边界案例：单条信号不能升级成趋势。
- 风险案例：48 小时时效窗口之外的旧内容必须排除。

这三个案例是学习用最小集合，不代表完整质量保障。后续扩展到 12 个案例时，保持 `4 个正常 + 4 个边界 + 4 个风险` 的结构。

## 运行方式

```bash
python3 evaluate_radar.py \
  --cases evals/cases.jsonl \
  --output evals/baseline-report.md
```

默认模式即使存在未通过案例也返回成功，因为它的作用是生成“现状基线”。修复已知缺口后，可增加 `--strict`，把同一套评测升级为发布门禁。
