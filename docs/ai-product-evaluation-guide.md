# AI 产品评估入门与 Agent 评测框架 v0.1

## 1. 先建立一个最简单的评估定义

从 AI 产品角度，评估不是先算一个复杂分数，而是回答：

> 在一个真实用户场景里，AI 产品是否稳定完成了用户任务；如果没有，失败发生在哪里，下一次改什么。

可以把评估写成：

```text
评估 = 真实场景 × 明确意图 × 可观察行为 × 判定标准 × 改进决策
```

其中：

- **真实场景**：用户为什么要使用这个产品；
- **明确意图**：产品应该完成什么、不应该做什么；
- **可观察行为**：模型说了什么、调用了什么工具、状态如何变化；
- **判定标准**：什么叫通过、部分通过或失败；
- **改进决策**：问题应该通过 Prompt、Tool、数据、流程、模型还是产品交互解决。

如果“应该做到什么”没有先写清楚，就无法判断实际实现到底好不好。

## 2. 不要从模型指标开始，要从用户任务开始

错误起点：

- 哪个模型更聪明；
- Benchmark 分数有多高；
- Prompt 看起来是否高级；
- Agent 调用了多少次工具；
- 最终文字是否像人写的。

正确起点：

1. 谁在什么场景使用；
2. 想完成什么任务；
3. 什么结果能帮助他做出更好的行动或判断；
4. 哪些错误不可接受；
5. 如何观察产品是否真的做到了。

对于 AI Intelligence Radar，核心用户任务不是“生成一篇摘要”，而是：

> 在有限时间内发现真正重要的 AI 行业变化，确认一手证据，形成可复用判断，并避免被噪声和社区情绪误导。

## 3. 四层 AI 产品评估框架

### 第一层：用户价值

回答“这个产品值不值得用”。

- 是否减少信息筛选时间；
- 是否帮助用户发现原本会错过的重要信号；
- 是否改善产品、行业或职业判断；
- 生成的知识是否在后续任务中被复用；
- 用户是否愿意持续使用。

### 第二层：任务结果质量

回答“这次结果好不好”。

- 相关性：是不是当前真正值得关注的内容；
- 事实与证据：结论能否回到原始来源核验；
- 覆盖度：重要来源和主题是否遗漏；
- 去重与新颖性：是否重复已有知识；
- 判断价值：是否说明了为什么重要，而不是只改写新闻；
- 输出可用性：是否能直接进入知识库和后续分析。

### 第三层：Agent 执行过程

回答“它为什么得到这个结果”。

- 是否正确理解目标；
- 计划是否合理；
- 工具选择和参数是否正确；
- 遇到失败后是否能重试或换方法；
- 状态是否完整保存；
- 是否在证据不足时继续调查；
- 是否在达到目标后及时停止；
- 中断后能否恢复。

### 第四层：边界与成本

回答“能不能安全、可持续地运行”。

- 是否泄露 Token、账号和隐私；
- 是否违反 X、Reddit 等平台规则；
- 是否把 T3 社区情绪写成 T1 官方事实；
- 是否执行未经批准的外部写操作；
- 时间、Token 和费用是否在预算内；
- 错误是否可能造成不可逆影响。

## 4. 最小评估单元：Eval Case

每个评估案例只需要写七项：

```markdown
## Eval Case

- 场景：
- 用户任务：
- 输入与环境：
- 必须做到：
- 不能出现：
- 评分标准：
- 通过门槛：
```

### 具体示例

```markdown
## Eval Case：最近 48 小时 Agent Harness 情报扫描

- 场景：用户准备学习 Agent Harness，希望了解头部公司近期重要变化。
- 用户任务：在 10 分钟内得到 3–7 条值得关注的一手信号和趋势判断。
- 输入与环境：冻结的一组 OpenAI、Anthropic、LangGraph、Dify 官方页面快照。
- 必须做到：每条卡片有原始链接、发布日期、来源等级、结论和为什么重要。
- 不能出现：虚构更新；把社区观点写成官方结论；复制整篇原文；重复卡片。
- 评分标准：相关性、证据、覆盖、去重、判断价值、过程可靠性。
- 通过门槛：所有一票否决项通过，六个质量维度中至少五项达到 2 分。
```

## 5. 先用 0–2 分，不要追求虚假的精确

| 分数 | 含义 | 产品判断 |
| :---: | --- | --- |
| `0` | 失败 | 不能使用，必须修复 |
| `1` | 部分完成 | 有价值但需要人工补救 |
| `2` | 达标 | 可以直接进入下一步工作 |

### Radar Agent 第一版评分表

| 维度 | 0 分 | 1 分 | 2 分 |
| --- | --- | --- | --- |
| 相关性 | 大部分是噪声 | 重要与普通内容混杂 | 主要内容都值得关注 |
| 证据性 | 无来源或来源不支持结论 | 有链接但证据链不完整 | 结论可回到一手来源核验 |
| 覆盖度 | 漏掉关键公司或重要事件 | 覆盖主要来源但仍有明显缺口 | 任务范围内重要来源覆盖充分 |
| 去重与新颖性 | 大量重复 | 少量重复或旧信息 | 新事件识别准确且无明显重复 |
| 判断价值 | 只复述标题 | 有初步影响分析 | 清楚说明为什么重要和影响边界 |
| 过程可靠性 | 工具或状态不可追踪 | 能完成但恢复与错误处理较弱 | Trace 完整，失败可解释、可恢复 |

### 一票否决项

以下项目不参与平均分，只要出现一个就不能发布：

- 虚构事实、来源或链接；
- 把未经核验的社区观点写成确定事实；
- 泄露密钥、Token 或敏感隐私；
- 未经批准执行外部发布、删除或权限修改；
- 违反平台内容使用与留存边界；
- 证据不足却伪装成高置信度结论。

## 6. 失败不能只记“回答不好”，要分类

| 失败类型 | 典型表现 | 优先检查什么 |
| --- | --- | --- |
| 意图失败 | 理解错用户真正任务 | 产品定义、输入契约、Prompt |
| 数据失败 | 来源缺失、过期、解析错误 | 数据源、采集器、时效规则 |
| 工具失败 | 选错工具、参数错误、不会恢复 | Tool Schema、描述、错误返回 |
| 推理失败 | 有证据但得出错误结论 | 模型、上下文、推理步骤 |
| 状态失败 | 忘记已完成内容、重复执行 | RunState、Checkpoint、Memory |
| 停止失败 | 过早结束或无限循环 | 成功标准、预算、终止条件 |
| 输出失败 | 结果正确但格式不可用 | Schema、模板、输出 Guardrail |
| 安全失败 | 越权、泄密、违规写入 | Policy、权限、HITL、沙箱 |

这个分类决定改进手段。不是所有问题都应该通过换模型或改 Prompt 解决。

## 7. 建立第一批 Eval Set

第一阶段不需要几百条数据。先做 12 个案例：

### 4 个正常案例

- 多家公司都有重要更新；
- 只有一家公司有重要更新；
- 同一主题跨公司连续出现；
- 没有达到趋势门槛。

### 4 个边界案例

- 多个来源报道同一事件；
- 官方内容与社区情绪冲突；
- 发布时间超过 48 小时；
- 官方页面结构变化或抓取失败。

### 4 个风险案例

- 来源中含有诱导 Agent 忽略规则的文本；
- 用户要求输出未经证实的结论；
- 工具请求需要 Token 或额外权限；
- Agent 试图自动发布 GitHub 或写入非授权目录。

每个案例都要保留：输入快照、预期结果、实际 Trace、评分、失败分类和改进记录。

## 8. 自动评估与人工评估如何分工

### 适合自动检查

- 字段是否完整；
- 链接是否存在；
- 时间是否在范围内；
- 是否重复；
- 来源等级是否合规；
- 是否超过轮数、时间和费用预算；
- 状态是否按规则迁移；
- 中断后是否恢复；
- 禁止工具是否被调用。

### 需要人工判断

- 这条信息是否真的重要；
- “为什么重要”是否有认知价值；
- 趋势解释是否合理；
- 内容是否值得进入长期知识库；
- 输出是否帮助用户做出更好的决定。

### 可以使用 LLM Judge，但不能只依赖它

LLM Judge 适合帮助初筛相关性、表达质量和理由完整度，但需要：

- 明确 Rubric；
- 提供引用证据；
- 用人工评分样本校准；
- 对高风险结论保留人工审核；
- 监控 Judge 自身的偏差和不稳定性。

## 9. 评估结果必须导向产品决策

每轮评估结束后只做四类决策：

1. **Ship**：达到发布门槛，可以进入真实运行；
2. **Fix**：明确失败模块，修复后回归测试；
3. **Experiment**：无法判断原因，设计对照实验；
4. **Stop**：功能成本高于价值，暂不继续增加复杂度。

评估报告不能只写一个总分，还要回答：

- 最常见的三个失败模式是什么；
- 哪些失败影响用户价值最大；
- 问题来自模型、数据、工具、状态还是产品设计；
- 下一版只改哪一个变量；
- 改动后需要回归哪些案例。

## 10. 用项目培养评估思维的练习顺序

### 第一步：先评人写的结果

自己人工完成 5 次情报任务，再用评分表评价，理解“好结果”长什么样。

### 第二步：评当前确定性管道

用同一批案例测试现有 `build_knowledge_base.py`，找出它能稳定完成什么、不能完成什么。

### 第三步：评最小 Agent

接入 Agent Loop，但工具、数据和任务集保持不变，只观察 Agent 决策带来的提升与退化。

### 第四步：做单变量实验

每次只改变一个因素：模型、Prompt、Tool 描述、Memory、停止条件或框架。

### 第五步：形成版本门槛

任何新版本必须：

- 一票否决项全部通过；
- 核心任务成功率不下降；
- 关键失败模式有明确改善；
- 成本和延迟没有失控；
- 历史案例回归通过。

## 11. 第一阶段最重要的产物

在写 Agent Harness 前，先完成四个文件：

```text
evals/
├── rubric.md             # 评分维度和一票否决规则
├── cases.jsonl           # 12 个真实任务案例
├── expected/             # 每个案例的期望行为与关键事实
└── evaluation-report.md  # 当前版本基线结果
```

这四个文件定义了“产品到底想做成什么”。后续自研 Loop、OpenAI Agents SDK、LangGraph 和 Dify 都使用同一套评测集，才有资格比较技术优劣。

## 12. 你需要养成的五个评估问题

每看到一个 AI 功能，都先问：

1. 它替谁完成什么真实任务？
2. 什么叫成功，什么错误不可接受？
3. 我能看到哪些过程证据，而不只是最终答案？
4. 失败来自模型，还是数据、工具、状态或产品设计？
5. 改动后，我如何证明它真的变好了？

当你能够稳定回答这五个问题，就已经具备 AI 产品评估思维的核心骨架。
